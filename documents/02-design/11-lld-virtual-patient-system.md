# 虚拟病人系统 — 技术规划与路线图

## 0) 目标

构建一个"虚拟病人" 3D 交互系统：
- 3D 扫描设备 → SMPL-X 参数重建 → 真实感渲染的数字化病人
- 医生/患者可在 3D 模型上标注、高亮身体部位进行讲解
- 整合患者体检数据、病史、用药记录，实现健康监控和风险预测
- 最终形态：医院诊室的智能数字人终端

## 1) 核心技术路径

```
┌──────────┐    ┌──────────────┐    ┌────────────────┐    ┌──────────────┐
│ 3D 扫描仪│ →  │ SMPL-X 拟合  │ →  │ GLB 真实感渲染 │ →  │ 浏览器 3D 展示│

                       │

               ┌──────────────┐

               │ EHR / FHIR   │
               └──────────────┘
```

## 2) 各层技术分析

### 2.1 3D 扫描 → SMPL-X 拟合

**状态**：项目已有 SMPL-X 模型权重、MoCap 动画管线、多源输入适配器

**路径**：
```
3D扫描仪输出（点云/深度图/RGBD）
  → 提取人体关键点（OpenPose / MediaPipe / YOLO-Pose）
  → SMPLify-X 拟合（已有 stub，需接入真实推理）
  → 输出 SMPL-X 参数（poses, betas, translation）
  → SMPL-X 模型生成 mesh（10475 顶点, 20908 面片）
```

**所需依赖**：
- `smplify-x`（GitHub: vchoutas/smplify-x）— 从关键点/点云拟合 SMPL-X
- `mediapipe` — 轻量级人体关键点检测
- `open3d` — 点云处理和配准
- `trimesh`（已有）— mesh 处理

**扫描仪兼容性**：SMPLify-X 支持从以下输入拟合：
- RGB 图片（最成熟）
- 深度图
- 稀疏关键点（OpenPose 格式）
- 点云（需自己实现 fitting loss）

**难度**：中高（SMPLify-X 有成熟代码，但需要针对扫描仪调参）

### 2.2 真实感人体渲染

这是整个系统最挑战的部分。分三个级别：

#### Level 1：基础 PBR 渲染（当前可用）
```
技术：Three.js MeshStandardMaterial / MeshPhysicalMaterial
效果：塑料/金属质感，肤色不自然
难度：低
```

#### Level 2：PBR 肤色材质（中期目标）
```
技术：
  - PBR 皮肤纹理贴图（diffuse + roughness + normal + specular）
  - Three.js MeshPhysicalMaterial + clearcoat
  - 环境光遮蔽（AO）
  - 可购买/下载专业皮肤纹理素材库
效果：看起来像游戏角色，比纯色几何体好很多
难度：中
```

#### Level 3：次表面散射 SSS（最终目标）
```
技术：
  - 皮肤次表面散射 shader（模拟光穿透皮肤的效果）
  - Three.js ShaderMaterial 自定义 GLSL
  - 参考实现：three.js 官方 SSS 示例
  - GPU 要求：支持 WebGL 2.0（所有现代浏览器）
效果：接近照片级真实
难度：中高
```

**关键参考项目**：
- Three.js SSS demo: examples/webgl_materials_subsurface_scattering
- Metahuman (Unreal Engine) 的品质无法在 Web 端达到
- 但在浏览器中可以实现 80% 的"看起来像真人"效果

**真实感渲染的关键要素**（按重要性排序）：
1. **好的纹理贴图**（60% 的真实感来源）— 高质量肤色 diffuse/normal/roughness 贴图
2. **灯光系统**（20%）— 三点光 + 环境光 + 轮廓光
3. **次表面散射**（10%）— 皮肤半透明感
4. **细节**（10%）— 眼睛反射、嘴唇湿润、头发细节

### 2.3 3D 身体标注/高亮

**实现方案**：

```
方案 A：顶点着色（推荐）
  - 根据 SMPL-X 的语义分割（身体部位 → 顶点索引映射）
  - 对选中部位的顶点应用 emissive/color 材质变更
  - 优点：精确到解剖结构；缺点：需要 SMPL-X 部位映射

方案 B：叠加几何体
  - 在目标部位添加半透明发光体（sphere/capsule）
  - 优点：简单；缺点：不精确

方案 C：UV 贴图叠加
  - 预渲染多组"高亮贴图"（肝脏区/心脏区/关节区等）
  - 动态切换贴图叠加层
  - 优点：灵活美观；缺点：需要美术资源
```

**SMPL-X 顶点部位映射**：
SMPL-X 顶点有固定顺序（10475 个），可以按解剖结构分组：
- 头部（约 2000 顶点）
- 躯干（约 3000 顶点）
- 左臂/右臂（约 2000 顶点）
- 左腿/右腿（约 3000 顶点）
- 手部/脚部

通过顶点索引范围可实现粗粒度的身体部位高亮。

**难度**：中

### 2.4 患者健康数据集成

**架构**：

```
┌─────────────────┐
│ 患者档案 JSON    │  ← 当前已有（patient_store.py）
│ (FHIR 兼容)      │
├─────────────────┤
│ 姓名 / 年龄 / 性别 │
│ 病史 (medical_history)│
│ 用药 (medications)│
│ 体检数据 (recent_exam)│
│ 症状 (symptoms)   │
│ 3D 身体参数       │  ← 扩展：SMPL-X betas (体型), scans
│ 风险评分          │  ← 扩展：ML 预测模型输出
└─────────────────┘
         │
         ▼
┌─────────────────┐     ┌──────────────┐
│ 前端 3D 可视化   │ ──→ │ 指标面板      │
│ 身体标记映射     │     │ 趋势图        │
│ 异常区域高亮     │     │ 风险预警      │
└─────────────────┘     └──────────────┘
```

**数据标准**：推荐参考 FHIR R4 标准（国际医疗数据交换标准）

**难度**：中（主要是数据模型设计，不涉及 AI）

### 2.5 健康监控与预测

**MVP 阶段**（可快速实现）：
- 基于规则的异常值告警（血压 > 140、血糖 > 7.0 等）
- 历史趋势可视化（Chart.js / Recharts）
- 风险评分（基于 Framingham / QRISK 等公开评分模型）

**进阶阶段**（需要数据和模型）：
- ML 模型预测疾病风险（需要标注数据集）
- 药物相互作用检查（已有 drug_kb 知识库）
- 体态异常分析（SMPL-X 姿态偏差检测）

**难度**：规则告警低，ML 预测中高

## 3) 分阶段路线图

### Phase 1：基础虚拟病人（当前 → 2 周）
```
[ ] 3D 模型标注系统：顶点 → 身体部位映射
[ ] 身体部位高亮/选中（方案 A：顶点着色）
[ ] 体检数据面板增强（趋势图 + 异常告警）
[ ] PBR 皮肤材质（Level 2：纹理贴图）
```

### Phase 2：讲解交互（2 → 4 周）
```
[ ] 医生讲解模式：点击身体部位 → 弹出病历信息
[ ] 部位动画：highlight pulse, zoom to body part
[ ] 关联数据展示：点击膝盖 → 显示骨关节炎历史 + X光意象
[ ] 语音驱动讲解：TTS + 同步身体部位高亮
```

### Phase 3：真实感渲染（4 → 8 周）
```
[ ] 次表面散射 shader（Level 3）
[ ] 高质量皮肤纹理采集/购买
[ ] 眼睛/头发/嘴唇细节
[ ] 表情 blendshape 接入（SMPL-X FLAME 模型）
```

### Phase 4：扫描仪接入（视硬件采购）
```
[ ] 扫描仪 API 对接
[ ] SMPLify-X 拟合管线部署
[ ] 扫描 → 虚拟病人一键生成
[ ] 扫描前后对比（体态变化追踪）
```

### Phase 5：健康预测（长期）
```
[ ] 规则引擎（药物相互作用、异常指标告警）
[ ] ML 模型集成（疾病风险预测）
[ ] 趋势分析和预警推送
```

## 4) 关键技术风险

| 风险 | 等级 | 缓解方案 |
|------|------|---------|
| 真实感渲染达不到"真人"效果 | 中 | 接受 80% 真实度定位，侧重纹理贴图 + 灯光 |
| SMPL-X 拟合精度不满足临床需求 | 中 | 先用 RGB 图片走通，扫描仪作为增强输入 |
| 浏览器性能不足以渲染 SSS | 低 | WebGL 2.0 足够，fallback 用 PBR |
| 扫描仪接口不兼容 | 中 | 采购前确认 API / SDK 开放性 |
| 体态预测缺乏训练数据 | 高 | 先用规则模型，逐步积累数据 |

## 5) 结论

**整体可行性：高**

核心链路（扫描→SMPL-X→渲染→展示→数据集成）每一步都有成熟的技术方案。最挑战的部分是真实感渲染，但在浏览器中达到"游戏级"的真实度是完全可行的。

建议从 **Phase 1（身体标注 + PBR 材质）** 立即开始，这是最能体现系统价值且技术风险最低的部分。
